Отсутствие централизованного, функционального информационного ресурса приводит к росту нагрузки на очные приёмы, дублированию запросов и снижению прозрачности власти.

**Целью** курсового проекта является разработка клиентской части информационной системы — официального сайта администрации Сергиево-Посадского городского округа, реализованного с использованием фреймворка Laravel и системы управления базами данных MySQL.

# 1 ОСНОВНАЯ ЧАСТЬ

## 1.1 Анализ предметной области

Администрация Сергиево-Посадского городского округа осуществляет управление муниципальным образованием, предоставляет информацию жителям и обеспечивает приём обращений граждан. Информационное обеспечение включает публикацию новостей, сведений о руководстве (глава, заместители), подразделениях, территориях, документах, финансах, вакансиях, экологии, а также приём обращений и антикоррупционных сообщений в электронной форме.

В рамках деятельности администрации можно выделить следующих **участников**:
- **Гражданин (неавторизованный пользователь)** — просмотр публичных страниц, регистрация;
- **Зарегистрированный пользователь** — подача обращений, просмотр своих обращений и ответов, подача антикоррупционных сообщений;
- **Сотрудник** — публикация новостей и статей, управление каруселью, редактирование раздела «Администрация» (глава, заместители), вакансий, просмотр обращений и антикоррупционных сообщений, направление ответов;
- **Администратор** — всё, что сотрудник, плюс управление пользователями и назначение ролей.

**Основные бизнес-процессы**:
- Регистрация и управление пользователями;
- Публикация новостей и статей по разделам;
- Управление контентом главной страницы (карусель, блоки «Полезные разделы», «Ещё на сайте»);
- Редактирование информации о главе и заместителях главы администрации;
- Управление вакансиями;
- Приём и учёт обращений граждан;
- Приём антикоррупционных сообщений и направление ответов;
- Публикация справочной информации (телефоны, участковые, УК, вакансии).

Анализ показывает, что при отсутствии автоматизированной системы данные ведутся в разрозненных источниках, что приводит к риску потери актуальности, ошибкам при учёте обращений, сложностям при формировании отчётности и повышенной нагрузке на персонал. Внедрение официального сайта на базе веб-приложения с централизованной базой данных является обоснованным и необходимым.

**Нормативно-правовая база:** деятельность ОМСУ в информационной сфере регулируется ФЗ № 8-ФЗ «Об обеспечении доступа к информации о деятельности государственных органов и органов местного самоуправления» и ФЗ № 59-ФЗ «О порядке рассмотрения обращений граждан Российской Федерации». Для СМИ применимы требования о регистрации (свидетельство Роскомнадзора) и указании возрастного ценза 18+. Эти требования учтены в футере сайта.

## 1.2 Цель и задачи разработки

**Цель разработки** — создание полнофункционального веб-сайта администрации Сергиево-Посадского городского округа с использованием Laravel и MySQL, обеспечивающего:
- централизованное хранение данных (новости, статьи, администрация, обращения, вакансии);
- разграничение прав доступа по ролям (гость, пользователь, сотрудник, администратор);
- удобный и адаптивный пользовательский интерфейс;
- защиту персональных данных (ФЗ-152): хэширование паролей, HTTPS, изоляция данных;
- расширяемость архитектуры (модульность, слабая связанность).

**Ключевые задачи:** разработка структуры БД; реализация архитектуры MVC; настройка аутентификации Laravel; реализация middleware для разграничения доступа; разработка интерфейсов для администратора, сотрудника и пользователя; реализация CRUD для статей, обращений, вакансий, администрации; обеспечение целостности данных (валидация, транзакции, внешние ключи).

## 1.3 Основание выбора технологий

В качестве серверной платформы выбран **фреймворк Laravel**, реализующий архитектурный паттерн MVC. Laravel обеспечивает: встроенную систему маршрутизации; ORM Eloquent для работы с БД; систему миграций; механизм аутентификации; поддержку middleware; защиту от CSRF; высокую масштабируемость.

**В качестве СУБД** выбрана **MySQL**, обеспечивающая поддержку реляционной модели, транзакционность, внешние ключи, индексацию.

**Клиентская часть** реализована с использованием HTML5, CSS3, JavaScript, шаблонизатора Blade.

Выбор указанных технологий обусловлен их распространённостью, стабильностью и соответствием задачам информационного портала муниципального образования.

# 2 ТЕХНОЛОГИЧЕСКАЯ ЧАСТЬ

## 2.1 Постановка задачи

В рамках курсового проекта требуется разработать **клиентскую часть** информационной системы — официальный сайт администрации Сергиево-Посадского городского округа. Система реализуется в клиент-серверной архитектуре: клиент (браузер) взаимодействует с сервером через HTTP/HTTPS; серверная часть (Laravel) обрабатывает запросы, управляет данными и обеспечивает безопасность.

**Функциональные требования:** хранение данных о пользователях и ролях; распределение ролей (администратор, сотрудник, пользователь, гость); управление новостями, статьями, каруселью, главой и заместителями администрации, вакансиями; приём обращений граждан с вложениями; приём антикоррупционных сообщений; ответы на обращения и сообщения; публикация справочной информации; формирование главной страницы и футера с юридическим текстом и ссылками.

**Нефункциональные требования:** многопользовательский режим; безопасность (аутентификация, CSRF, валидация, хэширование паролей); целостность данных (транзакции, внешние ключи); интуитивный и адаптивный интерфейс; масштабируемость архитектуры; кроссбраузерность (Chrome, Firefox, Edge, Safari).

## 2.2 Требования к программе

### 2.2.1 Требования к функциональным характеристикам

1. **Авторизация пользователей** — вход по email и паролю, сессия, защита от перебора, хэширование паролей (bcrypt), возможность восстановления через email.
2. **Разграничение прав по ролям (RBAC)** — администратор (полный доступ), сотрудник (контент, обращения, вакансии, администрация), пользователь (обращения, антикоррупция), гость (публичные страницы).
3. **Управление контентом** — CRUD для статей по разделам (новости, информация, финансы и т.д.), карусели, главы и заместителей администрации, вакансий.
4. **Приём обращений** — форма с полями ФИО, адрес, email, телефон, текст, вложение; сохранение с привязкой к пользователю; просмотр и ответы сотрудником.
5. **Приём антикоррупционных сообщений** — аналогично обращениям с ответами.
6. **Главная страница** — карусель, последние новости, форма обращений, блоки «Полезные разделы» и «Ещё на сайте».
7. **Футер** — на всех страницах: ссылки на важные разделы, юридический текст (© 2008–2024, СМИ, свидетельство Роскомнадзор, учредитель, 18+).
8. **Панель администратора** — управление пользователями, ролями.
9. **Панель сотрудника** — статьи, карусель, обращения, антикоррупция, вакансии, администрация.

### 2.2.2 Нефункциональные требования

- **Защита персональных данных (ФЗ-152):** шифрование при передаче (HTTPS), хэширование паролей, RBAC, аудит по необходимости.
- **Целостность данных:** реляционная СУБД, транзакции, валидация, внешние ключи.
- **Масштабируемость:** модульная структура, возможность кэширования, очередей.
- **Интуитивный интерфейс:** единообразная навигация, адаптивный дизайн, понятные формы.
- **Отказоустойчивость:** валидация, обработка ошибок, резервное копирование БД.
- **Кроссбраузерность:** валидный HTML5/CSS3, работа в современных браузерах.

### 2.3.1 Общая архитектура

Система включает три уровня: **клиентский** (браузер пользователя); **сервер приложений** (Laravel, PHP); **сервер базы данных** (MySQL). Запросы проходят через веб-сервер (Apache/Nginx) к Laravel, который обрабатывает маршрутизацию, контроллеры, модели и формирует HTML через Blade.

## 2.6.1 Описание функциональной структуры (IDEF0)

Методология IDEF0 используется для описания функций системы и их связей с входами, выходами, управлением и механизмами. Нумерация дуг по сторонам блока (как в задании: верх 2, низ 2, право 1, лево 3) интерпретируется следующим образом: **сверху** — управление (2 элемента), **снизу** — выходы (2 элемента), **справа** — управление/контроль (1 элемент), **слева** — входы (3 элемента). Ниже дано текстовое описание диаграмм; графическое исполнение выполняется в среде BPwin, draw.io или аналогах по стандарту IDEF0.

**Контекстная диаграмма A-0 (уровень 0)**  
Единственный блок: «Официальный сайт администрации Сергиево-Посадского городского округа».

- **Входы (слева, 3):** запросы пользователей (переходы по разделам, отправка форм); контент от сотрудников (тексты новостей, статей, данные об администрации, вакансии); нормативные и справочные документы для размещения.
- **Выходы (снизу, 2):** отображаемая информация для граждан (страницы разделов, новости, документы, ответы на обращения); учётные данные и отчёты для администрации (обращения, антикоррупционные сообщения, список пользователей).
- **Управление (сверху, 2):** требования законодательства (8-ФЗ, 59-ФЗ, регламенты); политика безопасности и правила размещения информации (в т.ч. 18+).
- **Механизмы (справа, 1):** программно-аппаратный комплекс (веб-сервер, PHP, Laravel, СУБД, браузер пользователя).

**Декомпозиция A0 (диаграмма A1)**  
Блок A-0 декомпозирован на подфункции (блоки A1–A5). Все **граничные** стрелки дочерней диаграммы соответствуют стрелкам родительского блока и кодируются по ICOM (Input, Control, Output, Mechanism); **внутренние** стрелки связывают выходы одних блоков с входами, управлениями или механизмами других.

**Граничные стрелки (ICOM-коды)**  
- **I1** — запросы пользователей (переходы по разделам, запросы страниц).  
- **I2** — исходный контент (тексты новостей, статей, данные об администрации, вакансии).  
- **I3** — нормативные и справочные документы для размещения.  
- **C1** — требования законодательства (8-ФЗ, 59-ФЗ, регламенты).  
- **C2** — политика безопасности и правила размещения информации (в т.ч. 18+).  
- **O1** — публикуемая информация для граждан (страницы, новости, документы, ответы на обращения).  
- **O2** — учётные и отчётные данные для администрации (обращения, антикоррупционные сообщения, пользователи).  
- **M1** — программно-аппаратный комплекс (веб-сервер, PHP, Laravel, СУБД, браузер).

**Блоки и их связь через ICOM**

- **A1** — «Предоставлять публичную информацию»  
  Входы: I1 (запросы пользователей); внутренний вход — контент для отображения (выход A4). Управление: C1. Выходы: O1 (HTML-страницы разделов). Механизм: M1.

- **A2** — «Принимать и учитывать обращения граждан»  
  Входы: данные формы обращения (ФИО, адрес, email, текст, вложение) — от границы или от A1 (форма на сайте). Управление: C1 (59-ФЗ). Выходы: запись в БД, уведомление пользователю (в O1); реестр обращений (в O2). Механизм: M1; внутренний механизм — сессия/роль пользователя (выход A5).

- **A3** — «Принимать антикоррупционные сообщения и формировать ответы»  
  Входы: email, текст сообщения. Управление: C1 (антикоррупционное законодательство). Выходы: запись в БД, ответ заявителю (в O1); реестр сообщений (в O2). Механизм: M1; внутренний механизм — сессия/роль (выход A5).

- **A4** — «Управлять контентом и пользователями»  
  Входы: I2 (исходный контент), I3 (документы); действия сотрудников/администратора. Управление: C1, C2. Выходы: обновлённый контент на сайте (внутренний выход → вход A1); обновлённые списки пользователей (в O2). Механизм: M1; внутренний механизм — сессия и роль сотрудника/администратора (выход A5).

- **A5** — «Аутентифицировать пользователей и разграничивать доступ»  
  Входы: учётные данные (регистрация, вход). Управление: C2 (политика безопасности). Выходы: сессия, роль пользователя (внутренние выходы → механизмы для A2, A3, A4). Механизм: M1.

**Внутренние связи между блоками (выход → вход/управление/механизм)**  
1. **A4 → A1** (выход–вход): обновлённый контент, произведённый A4, является входом для A1 (предоставление информации).  
2. **A5 → A2, A5 → A3** (выход–механизм): сессия и роль пользователя, произведённые A5, выступают механизмом для приёма обращений и антикоррупционных сообщений.  
3. **A5 → A4** (выход–механизм): сессия и роль сотрудника/администратора являются механизмом для управления контентом и пользователями.  
4. Выходы A2 и A3 (записи обращений и сообщений) формируют граничный выход O2; выход A1 (страницы) и уведомления от A2, A3 формируют O1.

При построении диаграммы в графическом редакторе граничные стрелки подписываются ICOM-кодами (I1, I2, I3, C1, C2, O1, O2, M1) у несвязанного конца; внутренние стрелки проводятся от правой (выход) или нижней (механизм) стороны блока-источника к левой, верхней или нижней стороне блока-потребителя согласно семантике IDEF0.

### 2.6.2 Диаграммы потоков данных (DFD)

DFD описывают потоки данных между процессами, внешними сущностями и хранилищами данных.

**Уровень 0 (контекстная DFD)**  
Внешние сущности: «Гражданин», «Сотрудник/Администратор». Один процесс: «Официальный сайт». Потоки: от Гражданина — запросы страниц, данные форм (обращения, антикоррупция); к Гражданину — страницы сайта, подтверждения приёма обращений; от Сотрудника — управление контентом и ответы на обращения; к Сотруднику — интерфейс кабинета и списки обращений.

**Уровень 1 (декомпозиция процессов)**  
Процессы: 1) «Показ публичных страниц» — чтение из хранилищ «Новости», «Статьи», «Администрация», «Вакансии» и т.д.; 2) «Обработка обращений» — приём формы → запись в «Обращения», уведомление; 3) «Обработка антикоррупционных сообщений» — запись в «Антикоррупционные сообщения»; 4) «Редактирование контента» — обновление хранилищ сотрудником; 5) «Аутентификация» — проверка по «Пользователи», создание сессии. Хранилища данных соответствуют таблицам БД (см. раздел 2.4 и главу 3). Потоки данных между процессами и хранилищами обозначаются стрелками с подписями (например, «Текст обращения», «Ответ», «Публикация»).

**Уровень 2 (при необходимости)**  
Детализация, например, процесса «Обработка обращений»: проверка входных данных → сохранение в БД → уведомление пользователя; процесс «Редактирование контента» — выбор раздела (новости/администрация/вакансии) → загрузка формы → сохранение в соответствующую таблицу.

### 2.6.3 Use Case (прецеденты использования)

Основные акторы и прецеденты:

- **Гость:** просмотр всех публичных страниц; регистрация и вход; переход к форме обращения (с перенаправлением на вход при необходимости).
- **Пользователь (авторизованный):** подача обращения; просмотр своих обращений и ответов; подача антикоррупционного сообщения; просмотр раздела «Вакансии» и др.
- **Сотрудник:** всё, что пользователь; публикация и редактирование новостей и статей; управление каруселью; редактирование раздела «Глава» и «Заместители главы»; управление вакансиями; просмотр обращений и антикоррупционных сообщений и направление ответов.
- **Администратор:** всё, что сотрудник; управление пользователями (создание, роли).

Диаграммы Use Case строятся в UML (например, в draw.io, PlantUML): акторы слева/справа, прецеденты — эллипсы, связи — ассоциации и include/extend при необходимости.

### 2.6.4 Структура базы данных (сводка)

Спроектированная реляционная база данных содержит следующие основные таблицы и связи (подробное описание полей — в разделе 2.4.2 и главе 4).

**Основные таблицы предметной области**

- **users** — пользователи: id, name, email, email_verified_at, password, remember_token, role_id (FK → roles), created_at, updated_at.
- **roles** — роли: id, name, slug, description, timestamps.
- **article_sections** — разделы статей (новости, информация, финансы и т.д.): id, slug, name, route_name, sort_order, timestamps.
- **articles** — статьи/новости: id, article_section_id (FK), user_id (FK, nullable), title, slug, excerpt, body, image, published_at, timestamps.
- **article_files** — вложения к статьям: id, article_id (FK), original_name, path, timestamps.
- **carousel_slides** — слайды карусели на главной: id, image, sort_order, timestamps.
- **administration_heads** — глава округа (одна запись): id, photo, title, description, timestamps.
- **administration_deputies** — заместители главы: id, photo, name, position, description, contacts, sort_order, slug, timestamps.
- **appeals** — обращения граждан: id, user_id (FK, nullable), fio, postal_address, email, phone, body, attachment, response, responded_at, responded_by (FK → users), consent, ip_address, timestamps.
- **anticorruption_reports** — антикоррупционные сообщения: id, user_id (FK, nullable), email, body, response, responded_at, responded_by (FK), ip_address, timestamps.
- **vacancies** — вакансии: id, user_id (FK, nullable), title, slug, body, published_at, timestamps.

**Служебные таблицы (Laravel/системные)**

- **sessions** — сессии пользователей.
- **password_reset_tokens** — токены сброса пароля.
- **cache**, **cache_locks** — кэш.
- **jobs**, **job_batches**, **failed_jobs** — очереди заданий.

**ER-диаграмма (текстовое описание)**  
Связи: User N:1 Role; Article N:1 ArticleSection; Article 1:N ArticleFile; Article N:1 User (автор); Appeal N:1 User (заявитель); Appeal N:1 User (responded_by); Vacancy N:1 User; AnticorruptionReport N:1 User, N:1 User (responded_by). Атрибуты сущностей соответствуют перечисленным полям таблиц. Графическая ER-диаграмма строится в нотации «сущность–связь».

# 3 РЕАЛИЗАЦИЯ ВЕБ-ПРИЛОЖЕНИЯ (КРАТКОЕ ОПИСАНИЕ)

### 3.1. Архитектура приложения

Приложение построено по MVC: маршруты (routes/web.php) связывают URL с контроллерами; контроллеры обращаются к моделям (Eloquent) и передают данные в представления (Blade). Для разграничения доступа используются middleware: auth, role (admin/employee), staff. Публичные разделы доступны без входа; форма обращений и личный кабинет — после аутентификации; кабинет сотрудника и администратора — по ролям.

### 3.2. Реализация ключевых разделов

**Главная страница (HomeController):** вывод карусели (carousel_slides), последних новостей (articles по разделу news), блока «Обращения граждан» с включением формы (appeals.partials.form), блоков «Полезные разделы» (ссылки на администрация, обращения, документы, финансы, справочная, экология) и «Ещё на сайте» (ссылки на округ, совет депутатов, АНК, антикоррупция, вакансии, информация). Футер вынесен в layout (шапка + футер) и содержит ссылки на важные страницы и юридический текст (© 2008–2024, СМИ, свидетельство Роскомнадзор, учредитель, 18+).

**Администрация (AdministrationController):** выборка administration_heads (первая запись) и administration_deputies (по sort_order); отображение в представлении administration.blade.php. Редактирование главы и заместителей — в кабинете сотрудника (Staff\AdministrationController): загрузка фото в storage/app/public/administration/, сохранение и обновление записей.

**Обращения граждан (AppealController):** форма с полями ФИО, адрес, email, телефон, текст, вложение, согласие на обработку данных; сохранение в appeals с user_id при авторизации. Просмотр своих обращений и ответов — на странице обращений для авторизованного пользователя. Ответы сотрудников (Staff\AppealController): поле response, responded_at, responded_by.

**Антикоррупция, АНК:** приём сообщений (AnticorruptionController) с сохранением в anticorruption_reports; страница АНК (appeals.ank) — статический контент с текстом об антинаркотической комиссии и ссылкой на раздел «Обращения граждан». Футер на всех страницах — единый блок в layouts/app.blade.php с навигацией и юридическим текстом.

**Новости и разделы статей:** унифицированный вывод по article_sections (ArticleController); управление статьями и файлами вложений — в Staff\ArticleController. Вакансии: публичный показ по slug (VacancyController); CRUD в Staff\VacancyController. Документы и финансы — контроллеры и представления по разделам с контентом и при необходимости привязкой к статьям.

### 3.3. Безопасность и валидация

Применяются: CSRF-токен во всех формах; валидация входных данных в контроллерах (request->validate); хранение паролей в хэшированном виде (Laravel Hash); ограничение доступа по middleware; при загрузке файлов — проверка типа и размера (image, max:5120 для фото). IP-адрес сохраняется для обращений и антикоррупционных сообщений.

**Уведомления.** Отображение session('success') и session('error') в виде блоков alert над основным контентом.

## 4.5 Версия для слабовидящих

### 4.5.1 Назначение и расположение

На сайте реализована **версия для слабовидящих** — специальный режим отображения, обеспечивающий комфортное восприятие информации пользователями с нарушениями зрения и пожилыми людьми. Переключатель режима размещён в полосе под шапкой сайта (блок «a11y-bar») на всех страницах. Кнопка «Версия для слабовидящих» доступна без авторизации и одинаково отображается на десктопной и мобильной версиях.

### 4.5.2 Реализуемые возможности

При включении режима применяются следующие изменения:

- **Увеличенный шрифт** — базовый размер текста увеличен до 1,35rem (примерно 21–22 px) для улучшения читаемости.
- **Высокий контраст** — чёрно-белая цветовая схема: текст чёрный (#000) на белом фоне (#fff), что обеспечивает коэффициент контрастности не менее 4,5:1 в соответствии с ГОСТ Р 56832 и WCAG 2.1.
- **Читаемый шрифт** — используется шрифт Arial без засечек.
- **Увеличенный межстрочный интервал** — line-height 1,7 для снижения зрительной нагрузки.
- **Подчёркивание ссылок** — все ссылки подчёркнуты и имеют чёрный цвет для однозначной идентификации.
- **Чёрно-белое оформление интерфейса** — кнопки, поля ввода, таблицы, карточки, выпадающие меню и пагинация оформлены в чёрно-белой гамме с чёткими границами.
- **Белый фон карусели** — область карусели на главной странице получает белый фон для единообразия восприятия.
- **Сохранение фотографий в цвете** — изображения (фото главы, заместителей, новостей, слайды карусели) не подвергаются обесцвечиванию и отображаются в исходных цветах для сохранения информативности визуального контента.

Состояние режима сохраняется в localStorage браузера и автоматически применяется при переходе между страницами. Кнопка меняет подпись на «Обычная версия» при включённом режиме.

### 4.5.3 Соответствие стандартам

Реализация ориентирована на требования:

- **ГОСТ Р 56832** (доступность цифрового контента, аналог WCAG 2.1);
- **Критерий 1.4.3** — контрастность текста не менее 4,5:1;
- **Критерий 1.4.4** — возможность увеличения размера текста;
- **Критерий 1.4.8** — настраиваемое визуальное представление текста;
- **Критерий 2.1.1** — доступность управления с клавиатуры (кнопка фокусируема и активируема клавишей Enter);
- **Критерий 4.1.2** — программно определимое назначение элементов (атрибуты aria-pressed, aria-label).

### 4.5.4 Значимость наличия версии для слабовидящих

Наличие версии для слабовидящих на официальном сайте органов местного самоуправления имеет важное значение по следующим причинам:

**Социальная инклюзия.** Сайт администрации является основным каналом информирования жителей о деятельности ОМСУ, приёме обращений, документах и услугах. Ограничение доступа к информации для граждан с нарушениями зрения противоречит принципам равного доступа к государственным и муниципальным услугам и создания безбарьерной среды.

**Нормативные требования.** Федеральный закон № 181-ФЗ «О социальной защите инвалидов в Российской Федерации» и Государственная программа «Доступная среда» предусматривают обеспечение доступности информации для людей с ограниченными возможностями здоровья. Реализация версии для слабовидящих является практическим шагом по исполнению этих требований в цифровой среде.

**Расширение аудитории.** Режим полезен не только людям с инвалидностью по зрению, но и пожилым пользователям, лицам с временным снижением остроты зрения, пользователям в условиях неоптимального освещения. Увеличение шрифта и высокий контраст повышают удобство работы с сайтом для широкого круга посетителей.

**Повышение эффективности обратной связи.** Граждане с нарушениями зрения получают возможность самостоятельно ознакомиться с информацией об обращениях, заполнить форму обращения или антикоррупционного сообщения без посторонней помощи, что снижает нагрузку на очные приёмы и повышает самостоятельность граждан.

Таким образом, версия для слабовидящих является неотъемлемым элементом доступного и социально ориентированного информационного ресурса органа местного самоуправления.

# 5 РУКОВОДСТВО ПРОГРАММИСТА

## 5.1 Структура проекта Laravel

- `app/Models` — модели (User, Article, Appeal, Vacancy, AdministrationHead, AdministrationDeputy и др.);
- `app/Http/Controllers` — контроллеры (HomeController, AdministrationController, AppealController, Staff\*);
- `routes/web.php` — маршруты;
- `resources/views` — шаблоны Blade (layouts, components, appeals, staff);
- `database/migrations` — миграции БД;
- `app/Http/Middleware` — middleware (AdminMiddleware, StaffMiddleware).

## 5.2 Маршруты

Публичные: `/`, `/news`, `/administration`, `/documents`, `/finance`, `/appeals/work`, `/appeals/ank`, `/reference`, `/ecology`. С auth: `/appeals` (форма, просмотр), `/appeals/anticorruption`. С admin: `/admin/*`. С staff: `/staff/*` (статьи, карусель, обращения, вакансии, администрация).

## 5.3 Связи моделей

User belongsTo Role; Article belongsTo ArticleSection, belongsTo User; Article hasMany ArticleFile; Appeal belongsTo User, belongsTo User (responded_by); Vacancy belongsTo User; AdministrationHead, AdministrationDeputy — самостоятельные сущности.

## 5.4 Миграции

Структура БД задаётся миграциями Laravel. Команды: `php artisan migrate`, `php artisan migrate:rollback`.

## 5.5 Безопасность

Хэширование паролей (bcrypt), защита от CSRF, ORM (защита от SQL-инъекций), middleware для проверки ролей.

```sql
CREATE TABLE users (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    email VARCHAR(255) UNIQUE NOT NULL,
    password VARCHAR(255) NOT NULL,
    role_id BIGINT UNSIGNED NULL,
    created_at TIMESTAMP NULL,
    updated_at TIMESTAMP NULL
);

CREATE TABLE administration_heads (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    photo VARCHAR(255) NULL,
    title VARCHAR(255) NOT NULL,
    description LONGTEXT NULL,
    created_at TIMESTAMP NULL,
    updated_at TIMESTAMP NULL
);

CREATE TABLE administration_deputies (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    photo VARCHAR(255) NULL,
    name VARCHAR(255) NOT NULL,
    position VARCHAR(255) NULL,
    description TEXT NULL,
    contacts TEXT NULL,
    sort_order SMALLINT UNSIGNED DEFAULT 0,
    slug VARCHAR(255) UNIQUE NULL,
    created_at TIMESTAMP NULL,
    updated_at TIMESTAMP NULL
);

CREATE TABLE appeals (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT UNSIGNED NULL,
    fio VARCHAR(255) NOT NULL,
    postal_address VARCHAR(255) NULL,
    email VARCHAR(255) NOT NULL,
    phone VARCHAR(255) NULL,
    body TEXT NOT NULL,
    attachment VARCHAR(255) NULL,
    response TEXT NULL,
    responded_at TIMESTAMP NULL,
    responded_by BIGINT UNSIGNED NULL,
    consent BOOLEAN DEFAULT FALSE,
    created_at TIMESTAMP NULL,
    updated_at TIMESTAMP NULL
);
```
